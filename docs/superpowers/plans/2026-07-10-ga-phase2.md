# LMCC GA Phase 2 Implementation Plan — Key Handoff Centerpiece

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** The cross-company key handoff as the PRD's P0 centerpiece: expiry countdown, Expired lifecycle, the five error states as real reachable screens, billing preview at the paste-confirm moment, and durable activity/security event logging.

**Architecture:** Key facts live on the connection (`configuration.lmccKeyCreatedAt`); expiry derives at display time (`lmccDisplay`). Errors are data-driven states in `LmccPasteKeyFlow` with one full-screen error component. A tiny global `activityLogSlice` records lifecycle + security events (Phase 3 renders them per connection).

**Tech Stack:** React 18, TS, Zustand 4, Vitest, Flywheel tokens.

## Global Constraints

- Error treatments exactly per PRD: invalid = inline; not-recognized = full-screen + security event; already-used = full-screen + link to connections; expired = full-screen + expiry date; wrong-account = full-screen and NEVER display the account id from the key.
- Expiry: 7 days from key creation. Expired carries no config/traffic/billing; card stays deletable.
- No copy promising real-time certainty. Statuses can lag.
- All demo error triggers must be reachable in the mock (special demo keys + demo buttons).

---

### Task 1: Expiry model + countdown util
- `src/utils/lmccDisplay.ts`: add `keyExpiryInfo(keyCreatedAt: string, now?: string): { expired: boolean; label: string }` — label "Expires in 6d 23h" / "Expires in 3h 12m" / "Expired Jul 17". `displayStatus` returns 'Expired' for LMCC + status Pending + expired key.
- Wizard AWS Max creation stamps `configuration.lmccKeyCreatedAt = new Date().toISOString()`.
- Handoff screen: countdown chip (dot + text, amber) next to the key; copy "Keys expire 7 days after creation — an expired connection simply reads Expired, never billed."
- Tests: countdown labels at 6d23h / 3h / expired boundary; displayStatus Expired derivation.
- Commit.

### Task 2: Paste-flow error taxonomy (5 states)
- `LmccPasteKeyFlow.tsx`: `KeyErrorState = null | 'invalid' | 'expired' | 'not-recognized' | 'already-used' | 'wrong-account'`.
- Full-screen error component (replaces the flow body) with per-state icon, plain-language explanation, next-step actions:
  - expired → "Generate a new key" guidance + expiry date shown; buttons: Back to start.
  - not-recognized → regenerate guidance; logs a security event on entry.
  - already-used → link to Connections list + "Create new connection".
  - wrong-account → "confirm the correct AWS account or regenerate" — account id NEVER rendered.
  - invalid stays inline on the field (existing panel).
- Demo reachability: keys `DEMO-EXPIRED`, `DEMO-USED`, `DEMO-WRONG-ACCOUNT`, `DEMO-NOT-RECOGNIZED` trigger states; keep existing demo buttons, add a compact "Demo an error" row.
- Commit.

### Task 3: Billing preview at the paste-confirm moment
- Confirm stage shows: metro/bandwidth/account from the key readout + estimated monthly rate (reuse BillingPreview's rate map via a small exported `estimateMonthlyRate(mbps)` in `src/utils/lmccBilling.ts`) + "Billing starts only when the connection is Live — never at key creation."
- Commit.

### Task 4: Activity + security event log (foundation)
- `src/store/slices/activityLogSlice.ts`: `activityEvents: ActivityEvent[]`, `logActivity(e: Omit<ActivityEvent,'at'|'admin'>)` — stamps ISO time + admin (from `att_nb_user` localStorage). `ActivityEvent = { at, admin, connectionId?, type: 'created'|'key-generated'|'key-accepted'|'live'|'expired'|'bandwidth-changed'|'deleted'|'security', message }`. Wire into useStore.
- Wire calls: wizard AWS Max create → 'key-generated'; paste-flow confirm success → 'key-accepted'; not-recognized error → 'security' ("Activation key rejected by AWS — possible reuse or tampering").
- Tests: slice appends with timestamp+admin; security event from error state.
- Commit.

### Task 5: E2E + docs + memory
- Browser: handoff shows countdown; each demo key reaches its error screen; wrong-account screen contains no 12-digit id; confirm shows billing preview; activity events appear in store after a create + a security demo.
- Full suite (baseline 46 pre-existing failures). Memory update. Commit.
