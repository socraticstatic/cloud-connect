# LMCC GA Phase 3 Implementation Plan — Detail Page GA

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** The connection detail page per the GA notes: machinery removed, read-only Peering 1–4 sets, MACsec assurance, term/renewal, last-mile view, activity history, needs-attention explainer.

**Architecture:** All LMCC blocks live in `ConnectionOverview` (already LMCC-aware) plus the existing `logs` tab for activity. `LMCCStatusPanel` loses every datacenter/device/port reference; the four-path structure appears ONLY as the neutral Peering sets.

## Global Constraints

- Never show device identity, datacenter names, physical ports, or internal channel ids. Peering sets labeled "Peering 1–4" are the single sanctioned four-path exposure.
- Peering sets are READ-ONLY (Micah's ruling; the subnet pencil is removed).
- Omit the MD5 BGP password. MTU 9001. Show both ASNs + the peer IP the customer points at.
- MACsec: read-only assurance, scope = the AT&T↔AWS interconnect, must not read as end-to-end.
- Last mile: view-only; type/status; "ordered through AT&T, not here"; visually separate from tier/health.
- Health blurbs stay calm; routine recovery is not an alarm.
