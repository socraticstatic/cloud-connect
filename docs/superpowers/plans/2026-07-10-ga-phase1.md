# LMCC GA Phase 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** The two-track status/health engine, GA vocabulary (Live), the hybrid hub auto-grouping (location + technology type, no wizard step), and the 20 Gbps tier fix — per the rulings in `docs/superpowers/specs/2026-07-10-lmcc-ga-notes-review.md`.

**Architecture:** A pure derivation module (`lmccStatusEngine`) implements the PRD's worked example; display helpers map LMCC connections to customer vocabulary everywhere. Hub becomes a store side-effect of `addConnection` keyed on `location::routeDomain`; the wizard's hub step is deleted.

**Tech Stack:** React 18, TS, Zustand 4, Vitest, Flywheel tokens.

## Global Constraints

- Status words: Pending, Provisioning, Live, Needs attention, Expired, Deleting, Deleted. Never "active" for LMCC surfaces. Status always dot + text.
- Health exists only while Live: full / reduced-healing / degraded. Health never changes status while traffic flows.
- Live requires BOTH providers confirmed; 3-of-4 during provisioning is still Provisioning.
- Route domains: VPN vs everything else — never share a hub (two route tables).
- Hubs are never created/named by users; rename only on Hub detail page.
- No copy promising real-time certainty.
- Bandwidth tiers: 1, 2, 5, 10, **20**, 50, 100 Gbps.

### Task 1: Status/health engine (pure, tested)

**Files:** Create `src/data/lmccStatusEngine.ts`, test `src/data/lmccStatusEngine.test.ts`.

**Produces:**
```ts
export type CustomerStatus = 'Pending' | 'Provisioning' | 'Live' | 'Needs attention' | 'Expired' | 'Deleting' | 'Deleted';
export type HealthPosture = 'full' | 'reduced-healing' | 'degraded';
export interface CustomerState { status: CustomerStatus; health: HealthPosture | null }
export interface EngineInput {
  provisioningStatus: 'key-generated' | 'key-accepted' | 'negotiating' | 'bgp-forming' | 'live' | 'failed' | 'deleting' | 'deleted';
  pathsUp: number;           // 0..4
  attConfirmed: boolean;
  awsConfirmed: boolean;
  keyCreatedAt: string;      // ISO
  now?: string;              // ISO, injectable for tests
}
export function deriveCustomerState(input: EngineInput): CustomerState;
export const STATUS_META: Record<CustomerStatus, { dotClass: string; blurb: string }>;
export const HEALTH_META: Record<HealthPosture, { label: string; blurb: string }>;
```

Rules (from the PRD worked example):
- key-generated: <7 days since keyCreatedAt → Pending; ≥7 days → **Expired**. Health null.
- key-accepted / negotiating / bgp-forming → Provisioning regardless of pathsUp (3/4 ready is NOT Live). Health null.
- live + both confirmed: pathsUp 4 → Live/full; 3 → Live/reduced-healing; 1–2 → Live/degraded; 0 → **Needs attention** (can't pass traffic), health null.
- live but a provider unconfirmed → Provisioning (both must confirm).
- failed → Needs attention. deleting → Deleting. deleted → Deleted.
- HEALTH_META blurbs: full = "All four paths healthy."; reduced-healing = "One path is repairing itself — no action needed."; degraded = "Multiple paths are down. Traffic is flowing; we are restoring protection." No real-time promises anywhere in blurbs (statuses "can lag reality").

Tests (write first, expect module-missing failure, then implement, then PASS):
pending fresh key; expired at exactly 7 days; provisioning at 3/4 with one provider confirmed; live/full; live one-down → status stays Live + reduced-healing; live two-down → degraded; zero paths → Needs attention; failed → Needs attention; deleting/deleted passthrough; health is null in every non-Live state.

Commit: `feat(lmcc): two-track status/health derivation engine`.

### Task 3: Hybrid hub auto-grouping in the store

**Files:** Modify `src/store/slices/hubSlice.ts` (add `ensureHubFor`), `src/store/slices/connectionSlice.ts` (call it from `addConnection`), extend `src/types/hub.ts` (`autoGrouped?: boolean; routeDomain?: 'internet' | 'vpn'`). Test `src/store/slices/hubGrouping.test.ts`.

**Produces:** `ensureHubFor(connection: Connection): string` on the store — returns hub id; finds hub where `hub.location === connection.location && (hub.routeDomain ?? 'internet') === domain(connection)`, else creates `{ id: 'hub-auto-<slug>-<domain>', name: location or '<location> (VPN)', location, locations: [location], routeDomain, autoGrouped: true, status: 'active', connectionIds: [] , links: [], vnfIds: [] }` (match existing Hub required fields — copy shape from sampleInfrastructure entries). Appends the connection id (dedup-safe).

`domain(connection)`: `connection.type` contains 'VPN' → 'vpn' else 'internet'.

`addConnection` (connectionSlice): after inserting the connection, call `get().ensureHubFor(connection)` — connections carry `location` (fallback: `configuration.location` / metro name; if no location at all, skip grouping).

Tests: creating a connection at a new location creates an auto hub named after the location; second same-location internet connection joins the same hub; a VPN connection at the same location gets a separate `(VPN)` hub; dedup on repeated add; connection without location doesn't throw.

Commit: `feat(hubs): auto-group connections by location + route domain`.
